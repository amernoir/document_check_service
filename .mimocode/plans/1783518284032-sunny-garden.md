# Plan: Document Check Service — Тестовое задание Backend

## 1. Разбор требований

### Must Have (обязательные по заданию)

- [ ] POST /api/checks — принимает multipart/form-data + program (federal/regional), возвращает результат
- [ ] GET /api/checks — список всех проверок (id, дата, программа, статус, количество документов)
- [ ] GET /api/checks/{id} — полный результат конкретной проверки
- [ ] Определение типа документа по имени файла: contract, specification, invoice, act
- [ ] Проверка комплектности: federal = договор + спецификация + счёт + акт; regional = договор + счёт + акт
- [ ] Валидация формата: только PDF, DOCX, JPG, PNG
- [ ] Валидация размера: max 20 МБ
- [ ] Warning для нераспознанных имён файлов
- [ ] Итоговый статус: approved / rejected / check_in_progress
- [ ] Хранение результатов в PostgreSQL
- [ ] Сохранение истории (каждая загрузка = новая запись)
- [ ] Docker Compose с PostgreSQL
- [ ] Pydantic валидация
- [ ] Alembic миграции
- [ ] pytest минимум 5 тест-кейсов
- [ ] README.md с запуском, тестами, переменными окружения, архитектурой
- [ ] .env.example

### Nice to Have (усиливает впечатление)

- [ ] HTTP статус-коды: 400 (невалидные данные), 404 (check не найден), 422 (валидация Pydantic)
- [ ] Корректный формат ответа POST /api/checks (check_id, status, status_label, reason, issues, documents, extracted, checked_at)
- [ ] Красивый README с архитектурной схемой (mermaid/ASCII)
- [ ] Покрытие edge cases в тестах (дубли имён, пустые файлы, несколько файлов одного типа)
- [ ] Миграция для индексов (по check_id для быстрых выборок)

### Можно не делать (если не хватит времени)

- [ ] Реальный парсинг PDF/DOCX для извлечения данных (заглушка OK)
- [ ] Версионность отдельных файлов (достаточно версий пакетов)
- [ ] Авторизация/разграничение прав (в задании упомянуто, но не в требованиях к API)
- [ ] Offline-работа
- [ ] CI/CD пайплайн
- [ ] Rate limiting / throttling

## 2. Архитектурное решение

### Структура директорий

```
document_check_service/
├── app/
│   ├── __init__.py
│   ├── main.py                  # FastAPI app factory, lifespan, CORS
│   ├── api/
│   │   ├── __init__.py
│   │   └── checks.py            # Роутеры: POST /api/checks, GET /api/checks, GET /api/checks/{id}
│   ├── schemas/
│   │   ├── __init__.py
│   │   └── check.py             # Pydantic модели: CheckResponse, CheckCreate, Issue, DocumentInfo
│   ├── services/
│   │   ├── __init__.py
│   │   ├── classifier.py        # Определение типа документа по имени файла (чистая функция)
│   │   ├── validator.py         # Проверка комплектности пакета (чистая функция)
│   │   ├── status_resolver.py   # Определение итогового статуса (чистая функция)
│   │   └── check_service.py     # Оркестрация: принятие файлов → классификация → валидация → статус → сохранение
│   ├── repositories/
│   │   ├── __init__.py
│   │   └── check_repository.py  # CRUD операции с Check/Document через SQLAlchemy
│   ├── models/
│   │   ├── __init__.py
│   │   └── check.py             # SQLAlchemy модели: Check, Document
│   ├── core/
│   │   ├── __init__.py
│   │   ├── config.py            # Settings через pydantic-settings (из .env)
│   │   └── storage.py           # Локальное хранилище файлов (upload/ folder)
│   └── db/
│       ├── __init__.py
│       └── session.py           # AsyncSession, get_db dependency
├── tests/
│   ├── __init__.py
│   ├── conftest.py              # Фикстуры: test client, test DB, temp files
│   ├── test_classifier.py       # Unit-тесты определения типа документа
│   ├── test_validator.py        # Unit-тесты проверки комплектности
│   ├── test_status_resolver.py  # Unit-тесты определения статуса
│   └── test_api.py              # Интеграционные тесты API
├── alembic/
│   ├── env.py
│   └── versions/
├── alembic.ini
├── Dockerfile
├── docker-compose.yml
├── requirements.txt
├── .env.example
└── README.md
```

**Почему именно так:**
- `services/classifier.py`, `validator.py`, `status_resolver.py` — **чистые функции**, не зависят от FastAPI/SQL. Легко покрываются unit-тестами без моков БД
- `services/check_service.py` — оркестратор, связывает всё вместе. Зависит от repository и storage, но не от API-слоя
- `repositories/` — изолируют SQLAlchemy от бизнес-логики
- `schemas/` — Pydantic модели для API, отделены от SQLAlchemy моделей

### Модели данных

#### Таблица `checks`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | Уникальный ID проверки |
| program | VARCHAR(20) | "federal" или "regional" |
| status | VARCHAR(20) | "approved" / "rejected" / "check_in_progress" |
| status_label | VARCHAR(100) | Человекочитаемый статус |
| reason | TEXT | Причина (если rejected) |
| document_count | INTEGER | Количество документов |
| created_at | TIMESTAMP | Дата создания |
| extracted_data | JSONB | Извлечённые данные (заглушка) |

#### Таблица `documents`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | |
| check_id | UUID (FK → checks.id) | |
| filename | VARCHAR(255) | Оригинальное имя файла |
| detected_type | VARCHAR(50) | contract / specification / invoice / act / unknown |
| file_path | VARCHAR(500) | Путь к файлу на диске |
| size_kb | INTEGER | Размер в КБ |
| created_at | TIMESTAMP | |

#### Таблица `check_issues`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | |
| check_id | UUID (FK → checks.id) | |
| level | VARCHAR(10) | "error" / "warning" |
| message | TEXT | Описание проблемы |

### Цепочка вызовов POST /api/checks

```
HTTP Request (multipart + program)
  ↓
checks.py (router) — принимает файлы + program
  ↓
check_service.process_check(files, program)
  ├── 1. Сохраняет файлы в uploads/ (storage.save_files)
  ├── 2. Для каждого файла: classifier.classify(filename) → detected_type
  ├── 3. validator.validate(documents, program) → list[Issue]
  ├── 4. status_resolver.resolve(issues) → status, status_label, reason
  ├── 5. check_repository.create(check_data, documents, issues)
  └── 6. Возвращает CheckResponse
  ↓
HTTP Response (200 OK)
```

## 4. Риски и решения по неоднозначностям

### 4.1 Определение типа документа по имени файла

**Неоднозначность:** Как матчить? По regex? По ключевым словам? С учётом регистра/транслита?

**Решение:** Кейс-Insensitive поиск по ключевым словам в имени файла (до расширения):
- contract: `договор`, `contract`, `дог`, `dog`
- specification: `спецификация`, `specification`, `спец`, `spec`
- invoice: `счёт`, `счет`, `счёт-фактура`, `invoice`, `сч`, `schet`
- act: `акт`, `акт_приемки`, `act`, `upd`, `упд`

**Обоснование:** Просто, покрывает русские и английские имена, не зависит от порядка слов. Regex избыточен — достаточно `in` проверки по lowercase.

### 4.2 Несколько файлов одного типа

**Неоднозначность:** Что делать если 2 файла определены как contract?

**Решение:** Брать первый найденный, добавить warning "Найдено несколько файлов типа X". Не блокировать, но предупреждать.

### 4.3 Пустой пакет / нет файлов

**Неоднозначность:** Возвращать 400 или обрабатывать как rejected?

**Решение:** 400 Bad Request — "Не загружено ни одного файла". Это валидация входных данных.

### 4.4 Извлечённые данные (extracted)

**Неоднозначность:** Реально парсить или заглушка?

**Решение (согласовано):** Заглушка — возвращаем фиксированные данные с пометкой в README, что AI-модуль подключается отдельно.

### 4.5 Хранение файлов

**Неоднозначность:** Файловое хранилище vs БД?

**Решение (согласовано):** Локальная папка `uploads/`. Файлы сохраняются с уникальными именами (UUID + оригинальное расширение).

### 4.6 check_in_progress статус

**Неоднозначность:** Когда именно check_in_progress? После загрузки до завершения? Или как промежуточный?

**Решение:** Поскольку у нас синхронная обработка (нет реального AI-модуля), check_in_progress не будет использоваться в текущей реализации. Статус сразу resolved. В README объяснить, что в продакшене check_in_progress появляется при асинхронной обработке AI-модулем.

### 4.7 Формат имени файла для определения типа

**Неоднозначность:** Берём только имя без расширения? Или всё имя файла?

**Решение:** Берём `Path(filename).stem` — имя без расширения, затем lowercase и поиск по ключевым словам.
